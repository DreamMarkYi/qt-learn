# 新人入门指南

欢迎加入！这份文档带你用 15 分钟搞清楚「这个项目是什么、由哪些部分组成、代码在哪」，
细节实现请跳到对应的专题文档（文末有索引）。

## 1. 项目是什么

一句话：**一个类似 VTube Studio 的虚拟直播软件**。

- **前端（本机客户端，C++/Qt）**：抓桌面 + 抓摄像头 → GPU 合成（叠加 Live2D 虚拟形象）
  → 本地预览 / 推流到直播平台。
- **后端（驱动虚拟形象的动作，Python）**：基于 **MediaPipe** 的人脸 / 人体关键点识别，
  把识别结果换算成 Live2D 参数，实时喂给前端驱动模型做表情和动作。后续会继续扩展手势识别等能力。

前后端目前是同机部署的两个进程，通过本地 TCP 通信（细节见第 5 节），不是必须要理解成
「服务器 - 客户端」架构。

## 2. 技术栈 —— 用到了什么

### C++ 主程序（`qt-learn` target）

| 类别 | 具体用到的东西 | 用来做什么 |
|---|---|---|
| 语言/构建 | C++17，CMake ≥ 3.16 | 项目本体 |
| UI/框架 | Qt 6（Widgets、Multimedia、MultimediaWidgets、OpenGL、OpenGLWidgets、Network） | 窗口 UI、摄像头/桌面采集、跨线程信号槽、TCP socket |
| 图形渲染 | OpenGL 3.3 Core（`QOpenGLFunctions_3_3_Core`）+ GLSL 着色器 | GPU 合成、Live2D 渲染 |
| 虚拟形象 | Live2D Cubism SDK（Core 预编译库 + Framework 源码，OpenGL 后端） | 加载 `.model3.json` 模型、播放 `.motion3.json` 动作、驱动模型参数 |
| OpenGL 扩展加载 | GLEW（静态编译） | Cubism 渲染器要用，Windows 系统自带的 opengl32 只导出 GL 1.1 |
| 纹理/图片 | stb_image.h | 纹理加载 |
| 推流 | 外部 `ffmpeg` 进程（`QProcess` 拉起，libx264 编码）| 把合成画面 + 音频编码推到 RTMP 服务器 |
| 音频路由 | VoiceMeeter + Voicemod（Windows 虚拟声卡） | 把变声人声和背景音混合成一路，喂给 ffmpeg 的 DirectShow 音频输入 |
| 网络 | `QTcpSocket`，自定义「长度前缀 + 数据」二进制协议 | 和 Python 后端交换摄像头帧 / 识别结果 |
| 并发 | `QThread` 独立渲染线程 + 固定间隔 `QTimer`（60fps）、双 PBO 乒乓异步回读 | 渲染与采集解耦、避免阻塞主线程/GPU 流水线 |

### Python 后端（`tools/pose_sidecar`）

| 类别 | 具体用到的东西 | 用来做什么 |
|---|---|---|
| 语言 | Python 3.12 / 3.13 | 后端边车进程 |
| 识别 | `mediapipe`（Tasks API：`PoseLandmarker` / `HolisticLandmarker`） | 人体 33 关键点识别，或身体+脸+双手（holistic）识别 |
| 图像处理 | `opencv-python` | JPEG 解码、可视化调试窗口（`--show`） |
| 数值计算 | `numpy` | 关键点数组、几何/角度计算 |
| 信号平滑 | One-Euro Filter（项目内手写实现） | 抑制关键点抖动，兼顾跟手性和稳定性 |
| 网络 | 标准库 `socket`，与 C++ 端相同的长度前缀协议 | 收摄像头 JPEG，回传 JSON 识别结果 |

## 3. 功能清单

**采集与合成（C++ 主程序）**
- 桌面画面捕获（`QScreenCapture`）
- 摄像头画面捕获，支持多摄像头设备切换（下拉框选择）
- 桌面 + 摄像头画中画 + Live2D 虚拟形象三路 GPU 合成，60fps 固定节拍渲染（渲染帧率与采集帧率解耦）
- 画中画开关：推流时是否叠加摄像头画面（实时生效）
- 本地预览（合成画面实时显示在主窗口）
- RTMP 推流（画面 + 系统混音音频），开始/停止推流按钮
- 渲染分步耗时统计（`FrameProfiler`，调试用）
- 独立的 Live2D 模型查看器工具 `live2d_viewer`（不参与合成/推流，单独调试模型显示效果）

**虚拟形象驱动（Python 后端 + C++ 消费端）**
- Live2D 模型加载与动作（motion）播放
- 人体姿态识别 → 驱动模型「身体左右摆」参数
- 面部表情/头部姿态识别 → 驱动模型眨眼（左右眼独立）、眼球朝向、嘴巴开合/形状、眉毛、脸颊、头部三轴角度等参数
- 关键点抖动平滑（One-Euro Filter，可调参数控制稳定性/跟手性）
- 两种识别模式可切换：`pose`（仅身体，更快）/ `holistic`（身体+脸+双手，用于表情场景对比）

**规划中 / 待扩展**
- 手势识别（比心、点赞、挥手等语义动作触发），见第 9 节

## 5. 数据流全貌

```
┌─────────────┐   TCP: JPEG 帧      ┌───────────────────────┐
│ C++ 主程序   │ ──────────────────▶│ Python pose_sidecar     │
│ (摄像头画面) │                     │ (MediaPipe 关键点检测) │
│              │◀────────────────── │                         │
└──────┬───────┘   TCP: JSON 关键点  └───────────────────────┘
       │ PoseBus（线程安全总线）
       ▼
┌─────────────────────────────────────────────┐
│ GlCompositor 渲染线程 @60fps                  │
│  - 桌面帧 + 摄像头帧 上传纹理                  │
│  - L2dLayer 读 PoseBus → 驱动 Live2D 参数 → 渲染│
│  - 合成一张画面 (frameReady 信号)              │
└──────────┬───────────────────────┬───────────┘
           │                       │
           ▼                       ▼
     UI 本地预览              FFmpegStreamer → RTMP 推流
```

关键点：
- **摄像头只被 C++ 独占**，Python 端不单独开摄像头，靠 C++ 把帧转发过去（见
  `tools/pose_sidecar/README.md` 里说的「拓扑 B」），避免两边抢设备。
- Python 端已经把 MediaPipe 的原始关键点/blendshape 换算成 Live2D 标准参数语义值
  （`PoseTypes.h` 里的 `FaceData`/`BodyData`），C++ 端不需要再做数学换算，直接赋值给模型参数。
- `PoseBus` 只保留「最新一份」数据，不排队、不留历史——驱动动画只关心当前状态。

## 6. 怎么跑起来

1. **先启动 Python 后端**（详见 `tools/pose_sidecar/README.md`）：
   ```bash
   cd tools/pose_sidecar
   python -m venv .venv && .venv\Scripts\activate
   pip install -r requirements.txt
   python pose_server.py --port 5066 --show
   ```
2. **再启动 C++ 主程序**（用 Qt Creator 或 CMake 直接构建 `qt-learn` target 即可；
   C++ 端连不上 Python 端会每 2 秒自动重连，所以先后顺序不严格）。
3. 主窗口里选摄像头、点「开启预览」能看到桌面+摄像头合成画面（含 Live2D 形象跟随动作）；
   点「开始推流」会调用本地 ffmpeg 推到 RTMP 地址（当前写死在 `DesktopCapturer::startPush`
   里，是临时联调用的测试地址，正式使用需要按需修改）。

## 7. 需要具备/补的相关知识

**C++ / 前端方向**
- Qt 基础：`QObject`/信号槽机制、跨线程队列连接（自动线程安全的关键）、`QThread` 多线程模型
- Qt Multimedia：`QScreenCapture`/`QCamera`/`QVideoSink`/`QVideoFrame`，尤其是像素格式（NV12 等）
- OpenGL：GLSL 着色器、FBO 离屏渲染、纹理上传、VAO/VBO、PBO 异步回读（本项目重度使用）
- Live2D Cubism SDK 基本概念：`.model3.json` 模型结构、Parameter（参数）、`.motion3.json` 动作、
  Core（二进制推理库）与 Framework（胶水层）的分工
- CMake：理解如何链接预编译库（Cubism Core）、源码库（GLEW/Framework）、头文件库（stb）
- 音视频/推流基础：RTMP 协议、H.264 编码、FFmpeg 常用参数、Windows 下 DirectShow 音频设备
  与虚拟声卡（VoiceMeeter/Voicemod）路由原理

**Python / 算法方向**
- MediaPipe Tasks API 的使用方式（`PoseLandmarker`/`HolisticLandmarker`）、关键点编号定义
- 关键点 → Live2D 参数的换算知识：坐标归一化、深度/可见度含义、简单几何角度计算、
  blendshape 与面部表情参数的对应关系、`solvePnP` 头部姿态估计思路
- One-Euro Filter 滤波原理与调参方法（`min-cutoff`/`beta` 对稳定性 vs 跟手性的影响）
- Socket 网络编程：长度前缀（length-prefixed）协议的封包/拆包思路

**通用**
- 直播推流生态基础概念（推流地址、码率/帧率/分辨率、RTMP 服务器）
- 后续做手势识别扩展，需要补充：常见手势的关键点判定逻辑（如指尖距离、角度阈值）或
  轻量手势分类模型的基础知识

## 9. 现状 & 接下来

- 前端合成推流管线已经跑通：桌面+摄像头+Live2D 三路合成、60fps 固定节拍渲染、推流到 RTMP。
- 后端目前是 MediaPipe **Pose**（身体 33 点）+ 可选 **Holistic**（身体+脸+双手）两种模式，
  已经把关键语义值换算好并做了 One-Euro 平滑。
- 计划中的下一步：在 `tools/pose_sidecar` 里继续扩展**手势识别**（比如比心、点赞、挥手等
  语义动作触发），协议和总线（`PoseBus`/`PoseTypes.h`）都已经预留了扩展空间，加新的数据类型
  按 `FaceData`/`BodyData` 的模式加即可。
